66 /*
67  NOTE: this is a *compatibility* header. Some functions may look wrong at
68  first, but they're only "as wrong" as they would be on Visual C++. Our
69  priority is compatibility
70 
71  NOTE: unlike most people who write inline asm for GCC, I didn't pull the
72  constraints and the uses of __volatile__ out of my... hat. Do not touch
73  them. I hate cargo cult programming
74 
75  NOTE: be very careful with declaring "memory" clobbers. Some "obvious"
76  barriers aren't there in Visual C++ (e.g. __stosX)
77 
78  NOTE: review all intrinsics with a return value, add/remove __volatile__
79  where necessary. If an intrinsic whose value is ignored generates a no-op
80  under Visual C++, __volatile__ must be omitted; if it always generates code
81  (for example, if it has side effects), __volatile__ must be specified. GCC
82  will only optimize out non-volatile asm blocks with outputs, so input-only
83  blocks are safe. Oddities such as the non-volatile 'rdmsr' are intentional
84  and follow Visual C++ behavior
85 
86  NOTE: on GCC 4.1.0, please use the __sync_* built-ins for barriers and
87  atomic operations. Test the version like this:
88 
89  #if (__GNUC__ * 10000 + __GNUC_MINOR__ * 100 + __GNUC_PATCHLEVEL__) > 40100
90  ...
91 
92  Pay attention to the type of barrier. Make it match with what Visual C++
93  would use in the same case
94 */
